# **Retrospective Workflow**

## **Purpose**

The Retrospective Workflow enables continuous organizational learning.

Its purpose is to systematically analyze completed work, identify successes and failures, extract lessons, and improve future execution.

Every major project, feature, deployment, incident, optimization effort, and strategic initiative must complete a retrospective.

The goal is not blame.

The goal is learning.

---

# **Workflow Owner**

Primary Supervisor:

CEO Agent

Supporting Supervisors:

* Engineering Supervisor  
* Product Supervisor  
* Operations Supervisor  
* Security Supervisor  
* QA Supervisor

Review Authority:

Executive Organization

---

# **Mission**

Convert organizational experience into organizational intelligence.

Every completed effort should make the organization smarter than it was before.

---

The workflow begins after:

* project completion  
* deployment completion  
* incident resolution  
* optimization completion  
* milestone achievement

Mandatory for:

* critical incidents  
* failed projects  
* major releases  
* architectural migrations  
* security events

# **Retrospective Loop**

Collect Evidence

↓

Analyze Results

↓

Identify Successes

↓

Identify Failures

↓

Determine Root Causes

↓

Extract Lessons

↓

Create Improvements

↓

Update Memory

↓

Assign Actions

---

# **Stage 1 — Evidence Collection**

Gather:

* plans  
* designs  
* implementation reports  
* deployment reports  
* metrics  
* incidents

Output:

Evidence Package

---

# **Stage 2 — Outcome Review**

Evaluate:

* objectives achieved  
* objectives missed  
* quality outcomes  
* business outcomes  
* technical outcomes

Output:

Outcome Assessment

---

# **Stage 3 — Success Analysis**

Identify:

* successful decisions  
* effective workflows  
* high-performing agents  
* strong collaboration patterns

Questions:

* What worked?  
* Why did it work?  
* Can it be repeated?

Output:

Success Analysis Report

---

# **Stage 4 — Failure Analysis**

Identify:

* missed objectives  
* delays  
* quality issues  
* deployment issues  
* security issues

Questions:

* What failed?  
* Why did it fail?  
* How could it be prevented?

Output:

Failure Analysis Report

---

# **Stage 5 — Root Cause Analysis**

Use:

* Five Whys  
* Fault Tree Analysis  
* Dependency Analysis  
* Timeline Review

Avoid:

* blame  
* assumptions  
* unsupported conclusions

Output:

Root Cause Report

---

# **Stage 6 — Lesson Extraction**

Convert findings into:

* standards  
* policies  
* workflows  
* architectural guidance  
* organizational knowledge

Output:

Lessons Learned Report

---

# **Stage 7 — Improvement Planning**

Create:

* corrective actions  
* process improvements  
* workflow updates  
* training requirements  
* documentation updates

Each action must have:

* owner  
* priority  
* due date  
* success metric

Output:

Improvement Plan

---

# **Stage 8 — Memory Integration**

Update:

executive.md

engineering.md

product.md

operations.md

security.md

qa.md

Store:

* lessons learned  
* recurring issues  
* successful patterns  
* improvement actions

Output:

Memory Update Report

## **Quality**

Was testing sufficient?

Did quality gates work?

✓ Lessons documented

✓ Improvements assigned

✓ Executive review completed
